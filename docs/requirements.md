# ClipScope 要件定義ドラフト v1

## 1. 目的

ClipScope は、Twitch 配信中に作成されたクリップを配信者本人が一覧で確認し、任意のクリップを OBS 上で再生できるようにする Windows 向け GUI アプリケーションである。

初期リリースでは、利用者の設定負担をできるだけ小さくし、OBS 側には再生用の Browser Source を 1 つ追加するだけで利用できる構成を目指す。

## 2. 対象ユーザー

- Twitch で配信を行う配信者本人
- OBS を利用して配信しているユーザー
- 配信中に作成されたクリップを後から確認し、配信演出や振り返りに使いたいユーザー

## 3. スコープ

### 3.1 初期リリースで対応すること

- Twitch アカウント認証
- ログインした Twitch ユーザー自身のチャンネルを監視対象にする
- Twitch クリップの定期ポーリングによる新規クリップ検出
- 取得したクリップの一覧表示
- GUI からクリップを選択
- OBS の Browser Source で選択したクリップを再生
- 基本的な設定保存

### 3.2 初期リリースで対応しないこと

- 複数チャンネル監視
- 他人のチャンネルを任意に指定して監視
- OBS の自動操作や obs-websocket 連携
- Twitch クリップ以外の動画ソース対応
- 高度な再生キュー制御
- クラウド同期

## 4. システム概要

本アプリは以下の構成を持つ。

- Python 製デスクトップ GUI アプリ
- Twitch API と連携する認証・取得機能
- ローカルで動作する簡易 Web サーバ
- OBS の Browser Source から参照される再生ページ

動作イメージ:

1. ユーザーがアプリで Twitch 認証を行う
2. アプリがログインユーザー自身のチャンネル情報を取得する
3. アプリが一定間隔でクリップ一覧を取得し、新規クリップを検出する
4. GUI にクリップ一覧を表示する
5. ユーザーが GUI 上で再生したいクリップを選択する
6. ローカル Web サーバ上の再生ページが選択中クリップを反映する
7. OBS の Browser Source に表示されたページでクリップが再生される

## 5. 機能要件

### 5.1 Twitch 認証

- 認証方式は Device Code Flow を採用する
- 配布アプリであるため、client secret をアプリに埋め込まない
- 認証後、アクセストークンおよび必要に応じてリフレッシュトークンを保持する
- トークン有効性の確認を行えること
- トークン失効時は再認証を促せること
- 初期リリースで要求する OAuth スコープはなしとする

### 5.1.1 OAuth スコープ方針

- 初期リリースでは Twitch API 呼び出しに必要な最小権限のみを要求する
- `Get Clips` は有効な OAuth トークンで呼び出せるため、追加スコープを要求しない
- ログインユーザー自身の特定には `Get Users` を利用する
- `Get Users` はメールアドレス取得を除き追加スコープ不要のため、追加スコープを要求しない
- よって、初期リリースで要求する OAuth スコープ一覧は空とする
- 将来、クリップ動画ファイルのダウンロードや編集系 API を利用する場合のみ追加スコープを再検討する

### 5.2 監視対象チャンネル

- 監視対象はログインした Twitch ユーザー本人の 1 チャンネル固定とする
- 初回認証後にユーザー情報を取得し、そのチャンネルを監視対象として確定する
- GUI から任意の別チャンネルを入力する機能は持たない

### 5.3 クリップ監視

- 監視方式は Twitch API のポーリングとする
- 一定間隔で Clip 一覧を取得し、新規クリップを検出する
- 監視開始時刻を基準時刻として保持し、`Get Clips` は `started_at` を指定して監視開始以降のクリップを取得する
- 初期リリースでは監視対象期間を「監視開始時刻から現在時刻まで」とし、配信開始前の過去クリップは取得対象外とする
- ポーリング間隔は設定可能にする
- 同一クリップの重複登録を防ぐ
- 取得失敗時は一定時間後に再試行する
- ネットワークエラーや API エラーが起きてもアプリ全体は停止しない

### 5.3.1 ポーリング間隔

- クリップ監視のポーリング間隔は設定可能とする
- 初期値は 10 秒とする
- 設定可能範囲は 5 秒から 60 秒までとする
- API 取得失敗時は通常ポーリングとは別に再試行制御を行う

### 5.4 クリップ一覧管理

- 取得したクリップをアプリ内で一覧管理する
- 一覧に最低限表示する情報は以下とする
- クリップタイトル
- 作成日時
- 作成者名
- クリップ URL
- サムネイル URL またはサムネイル画像
- 一覧は新しい順で表示できること
- 選択中のクリップが識別できること
- 既取得クリップの再取得時は一覧更新が破綻しないこと

### 5.4.1 クリップ保持方針

- 取得したクリップはローカルにメタデータとして保持する
- 初期リリースでの保持上限件数は 100 件とする
- 初期リリースでの保持期間は 30 日とする
- 保持件数または保持期間の条件を超えた場合、古いクリップから順に一覧対象外とする
- 動画ファイル自体はローカル保存しない
- 将来的に保持件数および保持期間は設定変更可能とする

### 5.5 クリップ再生

- GUI 上で選択したクリップを OBS 上で再生できること
- OBS との連携は Browser Source 1 個のみを前提とする
- Browser Source には、アプリ内ローカル Web サーバ上の固定 URL を設定する
- 再生対象の切り替えはアプリ側で行う
- 再生ページは現在の対象クリップ情報を取得し、表示内容を更新する
- OBS 用再生ページは Twitch Embed を含む最小構成の HTML とする
- 初期リリースでは iframe の `src` を JavaScript で動的に差し替えない
- クリップ切り替えは再生ページ全体の再読込によって行う
- 音声付き自動再生を優先し、ページ構成は自動再生成立を阻害しない形に保つ
- 完全なシームレス切り替えは必須要件としない

### 5.6 OBS 利用手順

- 利用者は OBS に Browser Source を 1 つ追加する
- Browser Source の URL はアプリが案内する固定ローカル URL を設定する
- 利用者に obs-websocket 設定を要求しない
- 利用者にクリップごとの URL 差し替えを要求しない

### 5.7 設定保存

- 以下の設定をローカル保存できること
- ポーリング間隔
- ローカル Web サーバのポート
- ウィンドウ状態
- 最後に使った認証情報の状態
- 起動時に保存済み設定を読み込めること

### 5.7.1 ローカル保存方式

- ローカル保存には SQLite を採用する
- 保存対象はクリップのメタデータおよびアプリ設定とする
- 動画ファイル自体はローカル保存しない
- クリップは `clip_id` を一意キーとして管理し、重複登録を防ぐ
- 保持上限件数 100 件および保持期間 30 日を超えたデータは、古いものから順に削除する
- 初期リリースでは単一ユーザー・単一端末での利用を前提とする

### 5.8 ログと状態表示

- 認証状態を確認できること
- 監視状態を確認できること
- 最終取得時刻を表示できること
- エラー発生時にユーザーが理解できる形でメッセージを表示すること
- 必要に応じて簡易ログを参照できること

## 6. 画面要件

### 6.1 初回セットアップ画面

- Twitch 認証開始ボタンを表示する
- Device Code および認証手順を表示する
- 認証完了状態を表示する
- OBS に設定する Browser Source 用 URL を表示する
- セットアップ完了後にメイン画面へ遷移できること

### 6.2 メイン画面

- 認証状態を表示する
- 監視状態を表示する
- 監視開始および停止を操作できること
- クリップ一覧を表示する
- 選択中クリップの詳細を表示する
- OBS 用 URL を確認できること
- 手動更新を実行できること
- 最終取得時刻および簡易ログを表示できること

### 6.3 設定画面

- ポーリング間隔を設定できること
- ローカルサーバポートを設定できること
- クリップ保持件数および保持期間を設定できること
- OBS 用 URL を確認できること
- Twitch 再認証を実行できること
- 各設定項目を初期値へ戻せること
- 設定全体を初期値へ戻せること
- 設定全体の初期化時は確認操作を要求すること

### 6.4 クリップ一覧 UI

- 1 行 1 クリップで表示
- タイトルと作成時刻が見やすいこと
- サムネイル表示はあると望ましい
- 選択操作が直感的であること

## 7. 技術要件

### 7.1 開発言語・ライブラリ

- 言語は Python
- GUI ライブラリは NiceGUI を第一候補とする
- HTTP 通信には一般的な Python HTTP クライアントを使用する
- ローカル Web サーバは NiceGUI もしくは同居可能な Web フレームワークで提供する

### 7.2 配布

- Windows 向け配布を前提とする
- 配布形態は単体実行ファイル化を想定する
- 初回利用者が Python を別途導入しなくても使える形が望ましい

### 7.3 ローカル再生ページ

- ローカル再生ページは固定 URL で配信する
- ページはアプリの現在状態を参照して再生対象を切り替える
- OBS 用再生ページは Twitch 埋め込み iframe をほぼ単独で含む最小構成とする
- 更新方式は短周期ポーリングで変更を検知し、ページ全体を再読込する方式を採用する
- 将来的に必要なら SSE などへの置き換えを検討する

### 7.3.1 Browser Source 用 URL 方針

- OBS の Browser Source には `http://localhost:<port>/obs-player` 形式の URL を設定する
- Twitch Clip 埋め込み時の `parent` パラメータには `localhost` を使用する
- 初期リリースでは `127.0.0.1` を既定値にしない
- ただし、環境差異に備えて `127.0.0.1` は検証候補として残す

## 8. 非機能要件

- 配信中に長時間起動しても安定動作すること
- 認証エラーや API エラーで異常終了しないこと
- 通常操作で利用者に複雑な設定を要求しないこと
- UI は配信中でも扱いやすいこと
- アプリ起動中の CPU / メモリ使用量は常識的な範囲に収めること

## 9. 制約・前提

- 対応プラットフォームは初期段階では Windows を前提とする
- Twitch API 利用には Twitch Developer 側のアプリ登録が必要
- OBS は Browser Source が利用可能であることを前提とする
- 利用者のローカル環境で指定ポートが使用可能である必要がある

## 10. リスク・要検証事項

### 10.1 Twitch クリップ埋め込み仕様

- Twitch Clip の埋め込みは iframe 前提であり、動画プレイヤー API のような細かな制御が難しい可能性がある
- クリップ切り替え時は iframe の再読み込みになる見込み
- 再生完了検知や細かなプレイヤー制御は難しい可能性がある
- 実機検証の結果、単純な iframe 埋め込みでは音声付き自動再生が成立した
- 一方で、JavaScript による iframe `src` の後付け変更や複雑な CSS / オーバーレイを含む構成では自動再生が不安定になることを確認した

### 10.2 parent / 埋め込み制約

- Twitch 埋め込みでは parent パラメータが必須
- ローカル URL での埋め込み動作可否を早期に検証する必要がある
- localhost と 127.0.0.1 のどちらを採用するかを早めに固定する必要がある
- HTTP レベルの事前検証では、`parent=localhost` および `parent=127.0.0.1` の埋め込み URL はいずれも 200 を返すことを確認済み
- `parent` を省略した場合は `embed-error.html?errorCode=NoParent` に遷移することを確認済み

### 10.3 Browser Source 上での動作

- OBS Browser Source 内で Twitch Clip 埋め込みが安定して再生されるか確認が必要
- 音声再生、自動再生、切り替え時挙動は試験対象とする
- 実機検証の結果、OBS Browser Source 上で音声付き自動再生およびクリップ切り替えが成立することを確認済み
- 成立条件は、OBS 用再生ページを最小構成に保ち、クリップ切り替え時はページ全体を再読込することである

### 10.4 API ポーリング設計

- クリップ作成直後に API へ反映されるまで遅延がある可能性がある
- ポーリング間隔が短すぎると API 利用効率が悪くなる
- ポーリング間隔が長すぎると GUI 反映が遅れる
- `Get Clips` は `broadcaster_id` のみ指定すると views 順で返るため、監視用途では `started_at` による期間指定を行う
- 取得時はページネーションを行い、対象期間内の取りこぼしを防止する

## 11. 初期リリースの成功条件

- Twitch 認証が完了する
- ログインユーザー自身のチャンネルクリップを取得できる
- 新規クリップを重複なく一覧に追加できる
- GUI から任意のクリップを選択できる
- OBS の Browser Source で選択クリップが表示・再生される
- 利用者が追加で高度な OBS 設定を行わずに利用できる

## 13. 内部アーキテクチャ

本アプリは以下の層で構成する。

### 13.1 GUI 層

- NiceGUI による画面表示とユーザー操作を担当する
- 初回セットアップ画面、メイン画面、設定画面を提供する
- Twitch API 呼び出しや SQLite 操作を直接持たず、アプリケーション層を通じて処理を実行する

### 13.2 アプリケーション層

- 認証、監視開始・停止、クリップ取得、再生対象変更、設定保存などの全体制御を担当する
- 各層の仲介および処理フローのオーケストレーションを担当する
- GUI 層からの操作要求を受け、各機能層へ処理を委譲する

### 13.3 Twitch 連携層

- Device Code Flow による認証処理を担当する
- トークン管理およびトークン有効性確認を担当する
- ログインユーザー情報取得およびクリップ一覧取得を担当する
- Twitch API との通信処理をこの層に集約する

### 13.4 クリップ管理層

- クリップ一覧の重複排除を担当する
- 一覧の並び順管理を担当する
- 保持上限件数および保持期間の適用を担当する
- 選択中クリップ状態の管理を担当する

### 13.5 OBS 再生ページ層

- ローカル Web サーバとして OBS Browser Source 向けページを提供する
- OBS 用再生ページは Twitch Embed を含む最小構成の HTML を返す
- クリップ変更時は iframe `src` の動的差し替えではなく、ページ全体の再読込で切り替える
- OBS 用再生ページの自動再生成立を阻害しないよう、余計な JavaScript や複雑な CSS を避ける

### 13.6 永続化層

- SQLite による設定、認証状態、クリップ情報の保存と読込を担当する
- クリップ情報の一意性管理および古いデータの削除を担当する
- アプリ起動時の設定復元を担当する

### 13.7 データフロー

- アプリ起動時に設定および認証状態を読み込む
- アプリ起動時に OBS 向けローカル Web サーバを起動する
- 監視開始後は Twitch 連携層が一定間隔でクリップを取得する
- 取得結果はクリップ管理層で整理した後、永続化層に保存する
- GUI 層は保存済みおよび最新状態のクリップ一覧を表示する
- ユーザーがクリップを選択した場合、アプリケーション層が選択中クリップ状態を更新する
- OBS 再生ページ層は選択中クリップ状態の変化を検知し、ページ全体の再読込によって再生対象を切り替える

## 14. 実装単位への分解

初期リリースでは、以下の実装単位で構成する。

### 14.1 想定パッケージ構成

- `app/main.py`
- `app/gui/`
- `app/application/`
- `app/twitch/`
- `app/clips/`
- `app/player/`
- `app/storage/`
- `app/models/`
- `app/config/`

### 14.2 エントリポイント

- `app/main.py` はアプリの起動点とする
- 設定読込、SQLite 初期化、ローカル Web サーバ起動、GUI 起動を担当する
- 各主要サービスの組み立てと依存関係の接続を担当する

### 14.3 モデル定義

- `app/models/settings.py`
- `AppSettings` を定義する
- ポーリング間隔、ローカルポート、保持件数、保持期間などを保持する

- `app/models/auth.py`
- `AuthState` を定義する
- アクセストークン、トークン状態、認証済みユーザー情報を保持する

- `app/models/clip.py`
- `ClipItem` を定義する
- clip_id、title、creator_name、url、embed_url、thumbnail_url、created_at、fetched_at などを保持する

- `app/models/playback.py`
- `PlaybackState` を定義する
- 現在選択中クリップ、最終切替時刻などを保持する

### 14.4 GUI 実装単位

- `app/gui/setup_page.py`
- 初回セットアップ画面を担当する
- Device Code 表示、認証案内、Browser Source URL 表示を担当する

- `app/gui/main_page.py`
- メイン画面を担当する
- 監視状態表示、クリップ一覧、選択中クリップ詳細、手動更新を担当する

- `app/gui/settings_page.py`
- 設定画面を担当する
- 各設定値の編集、初期値へ戻す操作、再認証操作を担当する

- `app/gui/components/clip_table.py`
- クリップ一覧表示用コンポーネントを担当する

### 14.5 アプリケーション層の実装単位

- `app/application/app_controller.py`
- アプリ全体の司令塔とする
- 起動処理、監視開始・停止、画面更新連携を担当する

- `app/application/monitor_service.py`
- ポーリング監視の開始・停止と定期実行を担当する
- Twitch からのクリップ取得を定期的に呼び出す

- `app/application/playback_service.py`
- 選択中クリップ状態の更新を担当する
- OBS 再生ページ向けの現在再生対象を管理する

- `app/application/settings_service.py`
- 設定の読込、保存、初期化を担当する

### 14.6 Twitch 連携層の実装単位

- `app/twitch/auth_client.py`
- Device Code Flow、トークン取得、トークン検証を担当する

- `app/twitch/api_client.py`
- Twitch API 呼び出しの共通処理を担当する
- ヘッダ付与、レスポンス処理、エラーハンドリングを担当する

- `app/twitch/user_service.py`
- ログインユーザー情報取得を担当する

- `app/twitch/clips_service.py`
- クリップ一覧取得を担当する

### 14.7 クリップ管理層の実装単位

- `app/clips/clip_repository_service.py`
- 取得済みクリップ一覧の整形と一覧更新を担当する

## 16. v0.1.3 追加要件

### 16.1 認証完了後の自動監視開始

- 未認証状態でアプリを起動した場合、ユーザーが認証を完了したタイミングで監視を自動開始する
- 自動開始対象はログイン済みユーザー本人チャンネルのみとする
- 監視開始に失敗した場合はアプリを停止させず、ユーザーに通知して手動開始へフォールバックする
- 既に監視中の場合は重複起動しない

- Windows 配布版（PyInstaller）でスプラッシュスクリーンを表示する
- `ClipScope.exe` 実行直後から、メインウィンドウが表示されるまでの待機時間を視覚化する
- スプラッシュはメイン UI 表示時に自動で閉じる
- スプラッシュ導入により既存の onefile 配布フローを維持する
- スプラッシュ画像や文言は将来的に差し替え可能な構成を目指す

### 16.3 受け入れ条件（v0.1.3）

- 未認証起動 → 認証完了後、監視ボタンが「監視を停止」状態になる（自動開始成功）
- 認証完了後の自動開始失敗時、アプリは操作可能なままエラーメッセージが表示される
- `ClipScope.exe` 起動時、メイン画面表示前にスプラッシュが表示される

## 17. v0.1.4 追加要件

### 17.1 自動再認証（リフレッシュトークン）

- `Get Clips` 実行前に認証状態を確認し、アクセストークン期限が近い場合は自動で更新する
- 対象は手動更新時と監視ポーリング時の両方とする
- API 呼び出しで 401 が発生した場合は、リフレッシュトークンで再認証して 1 回だけ再試行する
- リフレッシュ成功時は、新しいアクセストークンとリフレッシュトークンを保存する
- リフレッシュ不能（無効・期限切れ）時は未認証状態へ遷移し、再認証を促す

## 18. v0.1.5 追加要件

### 18.1 アップデート確認機能

- GitHub 上に配置した `version.json` を参照して、最新版の有無を確認できるようにする
- アプリ本体は自動更新しない
- 新しいバージョンが見つかった場合は、通知表示とダウンロードページを開く導線のみ提供する
- バージョン比較は `major.minor.patch` のセマンティックバージョン形式で行う
- ネットワーク未接続や取得失敗時でもアプリ本体の起動や操作を妨げない

### 18.2 `version.json` 配置方針

- `version.json` は GitHub で公開し、HTTP GET で取得できる固定 URL をアプリ内で参照する
- JSON には最低限、以下の情報を含める
  - `latest_version`
  - `download_url`
  - `release_notes_url`
  - `message`
- 形式は将来的な拡張を考慮し、追加フィールドがあっても無視できる構成とする

### 18.3 GUI 要件

- 設定タブの「一般」に、`自動更新確認` チェックボックスを追加する
- `自動更新確認` のデフォルト値は `True` とする
- 設定タブの「一般」に、`更新を確認` ボタンを追加する
- 設定タブの「一般」に、`Ko-fiで支援` ボタンを追加する
- `Ko-fiで支援` ボタン押下時は、既定ブラウザで `https://ko-fi.com/heconis` を開く
- `Ko-fiで支援` は設定値ではないため、保存対象にはしない
- 手動確認ボタン押下時は、その場で最新版確認を実行する
- 手動確認結果は GUI 通知で表示する
  - 最新版の場合: 「最新バージョンです」
  - 新版ありの場合: バージョン番号付きで通知し、ダウンロード導線を表示する
  - 取得失敗時: 簡潔な失敗通知のみ表示する

### 18.4 起動時動作

- `自動更新確認` が有効な場合、起動後にバックグラウンドで 1 回だけ最新版確認を行う
- 起動時確認は UI 表示をブロックしない
- 起動時確認の通信タイムアウトは短めに設定する
- 起動時確認で失敗しても、通常はエラー通知を出さず静かに失敗扱いとする

### 18.5 受け入れ条件（v0.1.5）

- GitHub 上の `version.json` が現行版と同じ場合、手動確認で「最新バージョンです」と表示される
- GitHub 上の `version.json` が現行版より新しい場合、新版あり通知とダウンロード導線が表示される
- `自動更新確認` を OFF にした場合、起動時確認は行われない
- 起動時確認失敗時でもアプリは通常どおり起動・操作できる

### 19.1 クリップ再生時間の表示
